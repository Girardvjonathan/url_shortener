# URL Shortener — Implementation Plan

An internal URL shortener built with **FastAPI + PostgreSQL + Redis + Jinja2**, designed for <100k URLs.

## Architecture Overview

```
                          ┌──────────────┐
    Browser ──────────────│   FastAPI     │
    POST /api/shorten     │   (uvicorn)  │
    GET /{short_code}     └──────┬───────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼             ▼
              ┌──────────┐ ┌────────────┐ ┌──────────────┐
              │  Redis   │ │ PostgreSQL │ │  Jinja2      │
              │ (Proxmox)│ │  (remote)  │ │  Templates   │
              └──────────┘ └────────────┘ └──────────────┘
```

**Cache-Aside Pattern** for redirects:
1. User hits `GET /{short_code}`
2. Check Redis for the short code → if **hit**, redirect immediately
3. If **miss**, query PostgreSQL → cache result in Redis (TTL: 24h) → redirect
4. Log the click asynchronously (don't block the redirect)

## Database Schema (PostgreSQL)

```sql
CREATE TABLE IF NOT EXISTS urls (
    id BIGINT PRIMARY KEY,
    short_code VARCHAR(10) UNIQUE NOT NULL,
    full_url TEXT UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS clicks (
    id SERIAL PRIMARY KEY,
    short_code VARCHAR(10) NOT NULL REFERENCES urls(short_code),
    clicked_at TIMESTAMPTZ DEFAULT NOW(),
    ip_address INET,
    user_agent TEXT,
    referrer TEXT
);

CREATE INDEX IF NOT EXISTS idx_short_code ON urls(short_code);
CREATE INDEX IF NOT EXISTS idx_full_url ON urls(full_url);
CREATE INDEX IF NOT EXISTS idx_clicks_short_code ON clicks(short_code);
CREATE INDEX IF NOT EXISTS idx_clicks_clicked_at ON clicks(clicked_at);
```

Key differences from a typical schema:
- `id` is `BIGINT`, **not** `SERIAL` — generated by the application, not the database
- `full_url` has a `UNIQUE` constraint — enables deduplication lookups

## URL Shortening Algorithm

**Flow:**
1. User submits `longURL`
2. Check if `longURL` already exists in the database
3. If **yes** → return the existing `shortURL` (deduplication)
4. If **no** → generate a new unique ID, base62 encode it, store and return

### Unique ID Generator

The ID is a **random integer** generated by the application (not auto-incremented
by the database). This makes short codes non-sequential and unguessable.

```python
import random

MAX_ID = 62 ** 3  # 238,328 — caps short codes at 3 characters

def generate_unique_id() -> int:
    return random.randint(0, MAX_ID - 1)
```

### Base62 Conversion

```python
CHARSET = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

def base62_encode(num: int) -> str:
    if num == 0:
        return CHARSET[0]
    result = []
    while num:
        num, rem = divmod(num, 62)
        result.append(CHARSET[rem])
    return ''.join(reversed(result))

def base62_decode(s: str) -> int:
    num = 0
    for char in s:
        num = num * 62 + CHARSET.index(char)
    return num
```

**Why cap at 3 characters?**
- 62³ = 238,328 possible IDs → short codes like `a`, `Z9`, `kX9`
- Sufficient for <100k URLs (~42% fill rate at capacity)
- Deliberately small keyspace to exercise collision handling in practice
- Example: ID `12345` → `3d7`, ID `0` → `0`, ID `238327` → `zzz`

> Note: To scale beyond 238k URLs, increase `MAX_ID` to `62 ** 4` (14.7M)
> for 4-character codes. No other code changes needed.

**Collision handling:**
1. Generate random ID in `[0, 238327]`
2. Attempt INSERT with `UNIQUE` constraint on both `id` and `short_code`
3. If unique violation → regenerate and retry (max 5 attempts)
4. At 100k URLs, ~42% of IDs are taken → expect retries on ~42% of inserts

#### [NEW] app/shortener.py
- `generate_unique_id() -> int` — generate a random integer in `[0, 238327]`
- `base62_encode(num: int) -> str` — convert integer to base62 string (1-3 chars)
- `base62_decode(s: str) -> int` — convert base62 string back to integer
- `create_short_url(pool, full_url) -> tuple[str, bool]` — deduplicate or create new, returns `(short_code, is_new)`
- `CHARSET = "0123456789ABCDEF...z"` — 62-character alphabet
- `MAX_ID = 62 ** 3` — caps output at 3 characters

#### [NEW] app/models.py
- `CreateURLRequest` — validates `full_url` (must be valid HTTP/HTTPS URL)
- `CreateURLResponse` — returns `short_url`, `full_url`, `short_code`, `created_at`

1. Check Redis cache for `short_code`

Create flow:
1. Check if `full_url` already exists in DB → return existing `short_code`
2. If new → generate random ID → base62 encode → INSERT (retry on collision)
3. Cache new mapping in Redis

## Decisions (Resolved)

- **Database**: PostgreSQL (user provides `DATABASE_URL` in `.env`)
- **Redis**: Running on user's Proxmox server (user provides `REDIS_URL` in `.env`)
- **Short codes**: Random ID → base62 encoded (max 3 chars, 238k keyspace), non-sequential
- **Deduplication**: Same longURL always returns the same shortURL
- **Base URL**: `http://localhost:8000` for development, `http://short.local` for production

## Verification Plan

### Automated Tests
1. **Unit tests** for `shortener.py` — verify base62 encode/decode roundtrips, ID generation range, collision retry logic
2. **Deduplication test** — submitting the same longURL twice returns the same shortURL
2. **API tests** using FastAPI's `TestClient`:
   - POST `/api/shorten` with valid URL → returns short code
   - POST `/api/shorten` with invalid URL → returns 422
   - GET `/{short_code}` → returns 301 redirect
   - GET `/{nonexistent}` → returns 404
3. **Cache tests** — verify Redis cache-aside behavior (hit/miss/fallback)

### Manual Verification
1. Start the app with `uvicorn app.main:app --reload`
2. Open the UI in a browser, create a short URL, verify redirect works
3. Check the recent URLs table updates with click counts
4. Verify Redis caching with `redis-cli GET <short_code>`
